## Interpretation

The successful selector change is maintenance-context recovery under strict
caps:

- maintenance summary-floor bypass requires direct evidence
- cleanup/refactor overflow is cheap and capped
- stronger maintenance candidates can replace weaker maintenance summaries
- deprecation cleanup has a separate content-evidence gate

The main remaining bottleneck is not cleanup/refactor. It is config/build:
`config_build` remains low recall and low token precision, so the next work
should start with diagnostics around config-only, source-only, and
config-plus-source tasks rather than a broad config cap increase.

## Caveats

- The benchmark command may report 128 public cases, but the scored release
  baseline has 108 rows with `expected_files`.
- Expected files are commit-diff labels, not exhaustive usefulness labels.
  Some selected "noise" is plausibly useful related context.
- Added files must be filtered out when benchmarking the parent checkout.
- Slice results are diagnostics. Release claims should use the full 108-case
  run above.
